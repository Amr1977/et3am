# Et3am Deployment Architecture

## Overview
Et3am uses a hybrid deployment approach:
- **Backend**: Automated via `deploy-auto.sh` (production servers)
- **Frontend**: Manual via Firebase (controlled CLI deployment)

## Issue Fixed (2026-04-04)

### Problem
The `deploy-auto.sh` script was **automatically deploying both backend and frontend** on every push to master. This caused the frontend to be overwritten with potentially outdated code, overriding manual Firebase deployments.

### Root Cause
- `deploy-auto.sh` was running on AWS and GCP machines
- It monitored master branch and deployed on every change
- Frontend build+deploy logic was included in the script
- Overwrote Firebase deployments made manually

### Solution
Removed the entire frontend deployment section from `deploy-auto.sh`. The script now **only handles backend deployment**.

## Current Deployment Flow

### Backend Deployment (Automatic)

**Trigger**: Every push to master branch  
**Handler**: `/home/ubuntu/et3am/deploy-auto.sh` (running via systemd/cron on production servers)  
**Process**:
1. Polls `origin/master` every 60 seconds
2. If new commits detected:
   - Git pull latest code
   - Check if backend commit changed in Firestore
   - If changed: `npm install` + `npx tsc` + `pm2 restart et3am-backend`
   - If not changed: Just reload via `pm2 restart`
3. Updates Firestore `deployments/backend` with commit hash

**Status**: ✅ Automatic - No manual intervention needed

### Frontend Deployment (Manual)

**Trigger**: Developer runs `npm run deploy` in frontend folder  
**Handler**: Firebase Hosting CLI  
**Process**:
1. Generate git-info.json (commit hash + date)
2. `npm run build` creates dist folder
3. `npm run deploy` → `firebase deploy --only hosting --project foodshare777`
4. Update Firestore `deployments/frontend` with commit hash (via frontend/src/firebase.ts)

**Status**: ✅ Manual - Full developer control  
**Why Manual?**
- Allows testing before deployment
- Prevents auto-overwriting with outdated code
- Independent from backend release cycle
- Can quickly roll back if needed

## Manual Deployment Scripts

### deploy.sh (Linux/Mac)
```bash
./deploy.sh                    # Deploy both frontend + backend
./deploy.sh --frontend-only    # Deploy only frontend
./deploy.sh --backend-only     # Deploy only backend
```

### deploy.ps1 (Windows PowerShell)
```powershell
.\deploy.ps1                   # Deploy both frontend + backend
.\deploy.ps1 -BackendOnly      # Deploy only backend
.\deploy.ps1 -FrontendOnly     # Deploy only frontend
```

## Deployment Status Tracking

Both backend and frontend track their deployed commits in Firestore:

**Collection**: `deployments`  
**Documents**:
- `backend`:
  ```json
  {
    "commit": "abc1234",
    "deployedAt": 1748934912345
  }
  ```
- `frontend`:
  ```json
  {
    "commit": "def5678",
    "deployedAt": 1748934968900
  }
  ```

This allows:
- Checking current deployed versions
- Detecting deployment mismatches
- Preventing re-deployments of same code

## Footer Version Display

Both frontend and backend can display their deployed versions:

**Frontend**: Shows in page footer
- Format: `commit: <hash> | <date> <time>`
- Generated by `scripts/generate-git-info.cjs` during build
- Updated with every `npm run build`

**Backend**: Available via API
- Future enhancement: `/api/version` endpoint

## Recommended Workflow

### For Backend Changes
1. Make changes in `backend/` folder
2. Push to master
3. `deploy-auto.sh` automatically deploys within 60 seconds
4. Monitor backend logs on server

### For Frontend Changes
1. Make changes in `frontend/` folder
2. **Test locally**: `npm run dev`
3. **Build locally**: `npm run build`
4. **Deploy manually**: `npm run deploy`
5. Check footer version matches your commit hash
6. Verify on live site

### For Both Backend + Frontend Changes
1. Push backend code to master
2. Wait for `deploy-auto.sh` to deploy backend (~60 seconds)
3. Test backend changes
4. Separately run `npm run deploy` for frontend
5. Verify both deployed (check footer + API)

## Rollback Procedures

### Frontend Rollback
1. In frontend folder, checkout previous commit:
   ```bash
   git log --oneline | head -5      # Find previous commit
   git reset --hard <commit-hash>   # Reset to that commit
   npm run deploy                   # Redeploy
   ```

### Backend Rollback
1. SSH to server:
   ```bash
   ssh ubuntu@api.et3am.com
   cd /home/ubuntu/et3am
   git reset --hard <commit-hash>
   cd backend && npm install && npm run build
   pm2 restart et3am-backend
   ```

Or let `deploy-auto.sh` handle it when code is pushed (simpler).

## Environment Files

- **Frontend**:
  - `.env.production` - Firebase project ID, API endpoint
  - Built into `dist/` and cached by Firebase CDN

- **Backend**:
  - `.env.production` - Database URL, secrets, API keys
  - Only exists on server, never committed to git
  - Loaded by `database.ts` based on NODE_ENV=production

## Security Notes

- Firebase token (`FIREBASE_TOKEN`) required for auto-deploy in CI/CD
- Git credentials must be configured on servers
- `.env` files should use SSH key or PAT authentication
- Secrets never committed to repository
