# Test Plan: BDD, Unit & E2E Tests for Et3am Food Donation Platform

## Overview
Create comprehensive test coverage including BDD feature files with step definitions, unit tests for backend API routes, and E2E tests for frontend workflows.

## Framework Stack (CONFIRMED)
- **BDD**: Cucumber.js with Vitest
- **Backend Unit Tests**: Vitest + Supertest
- **Frontend E2E Tests**: Playwright
- **Assertion Library**: @testing-library/react

## Implementation Steps

### Phase 1: BDD Feature Files

Create feature files in `backend/features/` and `tests/features/`:

1. **`user-authentication.feature`**
   - User registration with email/password
   - User login with credentials
   - Google OAuth login
   - JWT token validation

2. **`donation-management.feature`**
   - Create new donation (as donor)
   - View available donations
   - Reserve a donation (as receiver)
   - Complete delivery (as donor)
   - Cancel reservation
   - Daily limit enforcement for receivers

3. **`chat-system.feature`**
   - Send message during reserved status
   - Receive real-time messages
   - Read/unread message tracking

4. **`support-tickets.feature`**
   - Create support ticket
   - View own tickets
   - Admin responds to tickets

5. **`admin-dashboard.feature`**
   - View dashboard statistics
   - Manage users
   - Manage donations
   - View audit log

### Phase 2: Step Definitions

Create step definition files in `backend/src/steps/`:

1. **`auth.steps.ts`** - Steps for authentication scenarios
2. **`donation.steps.ts`** - Steps for donation CRUD operations
3. **`chat.steps.ts`** - Steps for chat interactions
4. **`support.steps.ts`** - Steps for support ticket flows

### Phase 3: Backend Unit Tests

Expand existing test coverage in `backend/src/routes/`:

1. **`donations.test.ts`** - Expand with reserve/complete/daily limit tests
2. **`auth.test.ts`** - JWT validation, middleware tests
3. **`chat.test.ts`** - Chat endpoint tests
4. **`support.test.ts`** - Support ticket tests
5. **`admin.test.ts`** - Admin panel tests
6. **`database.test.ts`** - Database operation tests

### Phase 4: Frontend E2E Tests (Playwright)

Create test files in `frontend/tests/e2e/`:

1. **`auth.spec.ts`** - Login/Register flows
2. **`donation-flow.spec.ts`** - Create, reserve, complete flow
3. **`chat.spec.ts`** - Real-time chat test
4. **`admin.spec.ts`** - Admin dashboard tests

### Phase 5: Component Unit Tests

Expand frontend component tests:

1. **`DonationCard.test.tsx`** - Already exists, expand coverage
2. **`Footer.test.tsx`** - Already exists

## Key Test Scenarios

### Daily Limit Test (Critical)
```
Given I am a receiver user
And I have not reserved any meal today
When I reserve a meal
Then the reservation should be successful
And my daily action count should be 1

Given I am a receiver user
And I have already reserved a meal today
When I try to reserve another meal
Then I should receive a daily limit error
But I should be able to complete the delivery (no limit on complete)
```

### Chat Test
```
Given a donation is in "reserved" status
When I send a message as donor
Then the receiver should receive the message in real-time
But when the donation is "completed"
Then I should not be able to send messages
```

